## GitHub 協作操作流程

### 首次設定（組長操作，W04 已完成）

1. 組長已用 Use this template 建立專題 Repo
2. 到 Settings → Collaborators → 邀請組員和教師 `pychang-ai`
3. 組員收到 GitHub 通知，點 Accept invitation

### 日常協作流程（全體組員）

每次開始工作前：
```bash
cd 你的專題資料夾
git pull origin main          # 先拉最新版本
```

完成工作後：
```bash
git add .
git commit -m "描述你做了什麼"
git push origin main          # 推上去
```

**重要**：先 pull 再 push，避免衝突。

### 遇到衝突時

如果 push 失敗顯示 `rejected`，表示其他人已經推了新的修改：

```bash
git pull origin main          # 先拉下來
# 如果有衝突，VS Code 會標示衝突的地方
# 手動修改衝突後：
git add .
git commit -m "解決衝突"
git push origin main
```

遇到不會解決的衝突，找組長或教師協助。

## 本週協作任務：繳交專題提案

### 步驟 1：組內討論（課堂 20 分鐘）

每位組員分享自己在 W04 探索的題目構想，全組討論後投票決定：

- 每人報告 2-3 分鐘：題目名稱、要解決什麼問題、可行性
- 全組討論各題目的優缺點
- 投票或共識決定最終題目

### 步驟 2：分工撰寫提案

決定題目後，打開 Repo 中的 `docs/主題企劃表.md`，分工填寫：

| 提案章節 | 建議負責人 |
|---------|---------|
| APP 名稱、問題描述 | 組長 |
| 目標使用者 | 組員 A |
| 核心功能清單 | 組員 B |
| 參考 APP、差異化 | 組員 C |
| 預計使用技術、分工 | 組員 D |

### 步驟 3：push 提案

每個人寫完自己負責的部分後：

```bash
git pull origin main                 # 先拉最新
# 編輯 docs/主題企劃表.md
git add docs/主題企劃表.md
git commit -m "填寫提案：[你負責的章節]"
git push origin main
```

組長最後確認所有章節都已填寫完整。

### 步驟 4：更新 GitHub Projects 看板

- 把「決定題目與繳交專題提案」卡片從 Todo 移到 Done
- 把「教師審查回饋後修正提案」卡片移到 In Progress

## Commit 訊息規範

每次 commit 請用以下前綴，方便追蹤：

| 前綴 | 用途 | 範例 |
|------|------|------|
| `docs:` | 文件更新 | `docs: 填寫專題提案目標使用者` |
| `feat:` | 新增功能 | `feat: 新增登入頁面` |
| `fix:` | 修正錯誤 | `fix: 修正閃退問題` |
| `style:` | 介面調整 | `style: 調整按鈕顏色` |
| `test:` | 測試相關 | `test: 新增單元測試` |

## 避免踩坑

| 常見問題 | 解法 |
|---------|------|
| push 被拒絕 | 先 `git pull`，解決衝突後再 push |
| 不小心改到別人的檔案 | 各人只改自己負責的部分，改之前先溝通 |
| 忘記 pull 就開始寫 | 養成習慣：每次打開專案先 pull |
| commit 訊息寫「update」 | 寫清楚做了什麼，方便追蹤 |
| 整組只有組長在 push | 每個人都要自己 push 自己的部分，這是個人貢獻的紀錄 |

## 教師查看方式

教師會從以下面向評估各組的協作狀況：

| 查看項目 | 來源 | 評估重點 |
|---------|------|---------|
| Repo 的 commit log | GitHub Insights → Contributors | 每位組員是否都有 commit |
| 主題企劃表完整度 | `docs/主題企劃表.md` | 各章節是否都已填寫 |
| GitHub Projects 看板 | Repo → Projects | 任務卡片是否有在移動 |
| 提案品質 | 教師 W06 審查 | 題目可行性、創新性、分工合理性 |
